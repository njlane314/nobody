nobody: a least-privilege runtime for autonomous software

This document is not a manual. It is a note on the architecture of a program whose job is to turn inherited authority into declared authority.

objectexecution boundary
interfacenobody run
first platformLinux
statusprototype, not sandbox

Abstract

nobody is a least-privilege runtime for agents, tools, MCP servers, and commands. It receives a subject to execute and a policy describing the authority that subject may exercise. It then constructs a confinement plan, starts the subject inside that boundary, mediates requests that cannot be handled by the kernel, and records the run as structured evidence.

The motivating defect is ambient authority: the tendency of a process to inherit the broad authority of the user who launched it. The design goal is to replace such inheritance with explicit capabilities over files, commands, network destinations, secrets, and tools.

Current status

The repository is a prototype. It parses policy, gates shell commands through an allow/deny list, runs allowed commands, and records JSONL trace events. It does not yet apply Landlock, seccomp, namespaces, cgroups, network mediation, MCP mediation, browser mediation, or secret brokering.

LayerCurrent repositoryIntended design
AuthorityPolicy file with shell allow/deny lists.Typed capabilities for files, processes, network, tools, secrets, and approvals.
EnforcementCommand gate before process launch.Kernel restrictions, mandatory proxies, overlays, and approval suspension points.
EvidenceAppend-only JSONL run trace.Versioned trace schema, replay, diff, audit export, and rule-level explanations.

1. Claim

An autonomous program should possess no authority not named in the policy for its current run.

This claim is stronger than logging. It is also narrower than general containerization. The relevant question is not whether a process is placed in a container, but which operations it is allowed to perform and whether the system can explain each decision.

A useful runtime must therefore make three things precise: the grant of authority, the mechanism that enforces the grant, and the evidence left behind after execution.

Claim partInspectable objectPrototype evidence
Declared authoritynobody.toml policy.Allowed and denied shell command lists.
Enforcement decisionRuntime allow/deny result.Denied command exits before execution.
Trace evidence.nobody/runs/latest.jsonl.Run start, policy decision, command result, and run exit events.

2. Problem

A developer shell is an authority-rich environment. It usually has access to source code, environment variables, SSH keys, cloud credentials, package registries, browsers, local sockets, internal networks, and collaboration tools. A conventional program launched from that shell inherits much of this authority without a separate act of delegation.

This model was imprecise for ordinary programs. It is more dangerous for agents. An agent can inspect a problem, generate commands, call tools, retry after failure, and combine local and remote actions. The human remains responsible for the consequences, but the human does not necessarily enumerate the intermediate actions.

The resulting failure mode is not merely malicious behavior. It is also ordinary error under excessive authority: reading the wrong file, leaking context to the wrong endpoint, invoking the wrong tool, running a destructive command, or using a secret in a context where the user did not intend it to appear.

3. Intent

The intended object is an execution boundary. nobody should sit between an agent and the machine, in the same sense that a shell sits between a user and programs. It should not decide how an agent thinks. It should decide what the agent is permitted to do. 

nobody run --policy policy.toml -- <agent-or-command>

The command line should remain small. The implementation behind it may be substantial: policy parsing, policy explanation, confinement, process supervision, filesystem mediation, network mediation, tool mediation, secret brokering, approvals, traces, diffs, and replay.

3.1 Compression target

The software compresses a stack that is normally spread across unrelated tools. The point is not to reimplement every feature of those tools. The point is to give agent execution a single, coherent authority model.

before sandbox constructor policy engine network proxy secret broker filesystem overlay + approval system + MCP gateway + audit log seccomp profile CI wrapper trace exporter diff viewer claim policy -> confinement -> execution -> trace surface nobody run --policy policy.toml -- <subject>

The useful analogues are small and composable systems: sandbox builders, profile-based confinement tools, Unix filters, policy decision points, and trace readers. The finished program should feel like a Unix tool, not a management console.

4. Design principles

4.1 No ambient authority

A subject receives no authority simply because the actor has it. Authority is passed through a policy and is scoped to a run.

4.2 Capabilities are typed

A grant to read a path is not a grant to write it. A grant to call one MCP tool is not a grant to call another. A grant to use a secret for a GitHub operation is not a grant to place it in the process environment.

4.3 Mechanism and decision are separate

Policy evaluation determines what should happen. Enforcement mechanisms determine how it is made true. Some decisions are implemented by the kernel, some by proxies, some by overlays, and some by approval gates.

4.4 Denial is a result, not an error

A denied request is normal operation. It should be recorded with the request, the matching rule, and the effect of the denial.

4.5 Fail closed

An invalid policy, unsupported primitive, failed proxy, failed secret broker, or unavailable approval path must not widen authority. The run either starts with its declared boundary or does not start.

4.6 Local first

The core runtime must work as a local binary. A hosted service may distribute policy and collect trace summaries, but it is not the source of the boundary.

4.7 Small trusted core

Code that changes privilege, creates namespaces, applies Landlock, installs seccomp filters, supervises the process tree, or routes secrets belongs to a small trusted core. User interfaces, model integrations, and enterprise workflows do not.

5. Model

The design uses a small vocabulary. This is intentional: a boundary that cannot be described simply will not be applied correctly.

TermMeaning
ActorThe human, CI job, service account, or organization principal that starts the run.
SubjectThe process tree, agent, command, MCP server, or tool being executed.
CapabilityA typed grant of authority over a resource or operation.
ResourceA file path, command, network destination, secret, tool, repository, or external system.
RequestAn attempted exercise of authority by the subject.
DecisionThe policy result for a request: allow, deny, or ask.
RunA bounded execution with actor, subject, policy, confinement, trace, and result.
TraceAn append-only sequence of events describing decisions and effects.

5.1 Authority relation

The policy evaluator is a total function over policy, request, and context. Its output includes a reason. The reason is part of the contract because the system must be auditable by inspection.

eval : Policy × Request × Context → Decision × Reason
DecisionRuntime interpretation
allowThe request proceeds. The trace records the matched rule and any observed effect.
denyThe request is blocked. The trace records the attempt, rule, and denial effect.
askThe request is suspended until an approval subject grants or denies it.

5.2 Run

The run is the central object. Prompts and chats are not sufficient units because they do not contain authority, confinement, side effects, or evidence. 

Run {
  id: RunId,
  actor: Principal,
  subject: CommandLine | AgentDescriptor,
  policy: PolicyDigest,
  confinement: ConfinementPlan,
  status: Running | Succeeded | Failed | Denied | Interrupted,
  trace: TraceRef,
  result: RunSummary
}

5.3 Capability grammar

Capability ::= FsRead(PathPattern)
             | FsWrite(PathPattern)
             | FsDelete(PathPattern)
             | ProcessExec(CommandPattern)
             | NetConnect(DestinationPattern)
             | ToolCall(Server, Tool, ArgumentConstraint)
             | SecretUse(SecretName, Sink)
             | Approval(OperationPattern)

6. Architecture

The architecture separates policy, enforcement, and evidence. These concerns interact, but they should not be fused. Fusing them would make policy harder to explain, enforcement harder to audit, and traces harder to trust.

policy file │ ▼ parse / validate │ ▼ normalize │ ▼ confinement plan │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ filesystem process network boundary boundary boundary │ │ │ └──────┬──────┴──────┬──────┘ ▼ ▼ tools/MCP secrets │ │ └──────┬──────┘ ▼ trace │ ▼ diff / replay / audit

6.1 Subsystems

A production repository should be a Rust workspace. The names are less important than dependency direction. The policy crate must not depend on the Linux confinement crate. The trusted core must not depend on the web interface.

nobody-cliCommand parsing, preflight, local interaction.
nobody-policyAST, validation, normalization, decision explanation.
nobody-runnerRun lifecycle, process supervision, cancellation.
nobody-linuxLandlock, namespaces, seccomp, cgroups.
nobody-fsPath matching, overlays, diffs, file event interpretation.
nobody-netDNS policy, egress proxy, network event capture.
nobody-mcpTool schema inspection, tool policy, protocol proxying.
nobody-secretsSecret providers, scoped injection, redaction.
nobody-traceEvent schema, append-only writer, readers, exporters.

6.2 Trusted core

The trusted core is the set of code whose failure can widen authority. It should be dependency-light, testable in isolation, and written so that unsafe operations are scarce and named. Code outside the trusted core may be inconvenient when wrong. Code inside it is dangerous when wrong. 

trusted core
  policy normalization
  confinement plan construction
  Linux restriction application
  process spawning and supervision
  secret routing
  trace durability for decisions

not trusted core
  marketing pages
  hosted dashboard
  model provider adapters
  report formatting
  documentation UI

7. Mechanisms

No single Linux primitive implements the whole model. The runtime must combine mechanisms and be explicit about what each one protects.

BoundaryMechanismsDesign note
FilesystemLandlock, mount namespaces, overlays, path policyPrefer kernel denial for paths; use overlays to separate proposed writes from committed writes.
ProcessUser, PID, and mount namespaces; seccomp; cgroupsRestrict descendants as well as the initial process.
NetworkNetwork namespace, DNS mediation, egress proxyMake arbitrary egress impossible under deny-default policy.
ToolsMCP proxy, schema validation, argument constraintsTreat a tool call as a capability-bearing operation.
SecretsBrokered retrieval, scoped injection, redactionDo not pass the actor's environment wholesale to the subject.
ApprovalSuspension point, approver identity, explicit decisionPause at irreversible edges rather than relying on post-hoc review.

7.1 Enforcement classes

Every capability should be assigned an enforcement class. This prevents the policy from promising more than the runtime can provide.

ClassMeaningExample
kernelDenied by operating-system primitives.Landlock blocks a denied file read.
proxyDenied by a mandatory user-space mediator.MCP proxy denies pull_request.merge.
overlayAllowed in a reversible write layer.Agent writes a proposed patch.
approvalSuspended until an explicit decision.Agent requests a production deploy.
observedRecorded, but not yet enforceable.Prototype-only trace of a command.

8. Policy

Policy is the source of authority. It must be readable, diffable, and explainable. A policy language that cannot be reviewed in a pull request is too heavy for the first system. 

name = "coding-agent"

[fs]
read = ["./", "~/.cargo/registry"]
write = ["./src", "./tests", "Cargo.toml", "Cargo.lock"]
deny = [".env", "~/.ssh", "~/.aws", "/var/run/docker.sock"]

[net]
default = "deny"
allow = ["github.com", "api.github.com", "crates.io", "api.anthropic.com"]

[shell]
allow = ["git", "cargo", "rustc", "python"]
deny = ["ssh", "scp", "curl", "sudo", "rm"]

[mcp.github]
allow_tools = ["repo.read", "pull_request.create"]
require_approval = ["pull_request.merge", "workflow.dispatch"]

[secrets]
default = "deny"

[trace]
mode = "full"

The important interface is not only run. It is explain. A user should be able to ask why a request would be allowed or denied without executing the subject. 

$ nobody policy simulate policy.toml fs.read ~/.ssh/id_ed25519
DENY  matched fs.deny "~/.ssh"

$ nobody policy simulate policy.toml net.connect github.com:443
ALLOW matched net.allow "github.com"

9. Trace

The trace is the evidence layer. It is not merely diagnostic output. It is the object used to inspect a run, justify a decision, reconstruct changes, export audit evidence, and debug the boundary itself. 

{"kind":"run.start", "run":"run_01J...", "policy":"sha256:..."}
{"kind":"policy.decision", "request":"fs.read", "path":"~/.ssh/id_ed25519", "decision":"deny", "rule":"fs.deny"}
{"kind":"net.connect", "host":"github.com", "port":443, "decision":"allow"}
{"kind":"mcp.tool_call", "server":"github", "tool":"pull_request.create", "decision":"ask"}
{"kind":"approval.granted", "approval":"appr_01J...", "actor":"njlane314"}
{"kind":"run.exit", "code":0, "status":"succeeded"}

A trace event should record four fields of interest: what was requested, what decision was made, why that decision was made, and what effect followed. If one of these cannot be recorded, the event schema should say so explicitly.

9.1 Replay

Replay does not initially mean deterministic re-execution of arbitrary programs. It means reconstruction of the authority history of a run. The minimum useful replay answers: what was allowed, what was denied, what changed, what external calls were made, and who approved irreversible actions.

10. Interface

The primary interface is the command line. The command line is not incidental; it is the natural place for a local execution boundary. It composes with shells, CI, editors, and other tools. 

$ nobody run --policy policy.toml -- claude-code

The preflight output should make authority visible before execution. The post-run output should summarize evidence after execution. The UI should be sparse: no decorative status, no anthropomorphic agent state, no vague risk meter without rule-level explanation. 

filesystem
  read   ./, ~/.cargo/registry
  write  ./src, ./tests, Cargo.toml
  deny   .env, ~/.ssh, ~/.aws, /var/run/docker.sock

network
  allow  github.com, api.github.com, crates.io
  deny   *

shell
  allow  git, cargo, rustc, python
  deny   ssh, scp, curl, sudo, rm

A web interface may be useful for organizations, but it should not become the conceptual center. The center is the run and its trace.

11. Threat model

The design should be honest about the adversary. nobody is not a proof that an arbitrary host is safe. It is a boundary that reduces the authority of a subject under a stated policy.

In scope

  • Agent attempts to read denied files.
  • Agent attempts arbitrary network egress.
  • Agent runs a denied command.
  • Agent calls an unauthorized MCP tool.
  • Agent requests a secret for an unapproved sink.
  • Agent proposes an irreversible external action.

Out of scope initially

  • Compromised host kernel.
  • Malicious root administrator.
  • Hardware side channels.
  • Perfect data-flow tracking through arbitrary process memory.
  • Full deterministic replay of all programs.
  • All browser GUI semantics.

The practical test is whether an organization can allow useful agent work while denying common accidental and intentional overreach. The test is not whether the runtime solves every confinement problem in computer security.

12. Enterprise form

The enterprise system is not a different product. It is coordination around the same local primitive. The local runtime enforces. The control plane distributes policy, stores approvals, summarizes traces, integrates identity, and exports evidence.

developer laptop CI runner hosted runner │ │ │ ▼ ▼ ▼ local nobody runtime local nobody runtime managed runtime │ │ │ └──────── signed trace summaries ────────────┘ │ ▼ enterprise control plane │ ┌────────────────────┼────────────────────┐ ▼ ▼ ▼ central policy approvals audit search identity / RBAC secret routing SIEM export

The enterprise value is permission to adopt agents without granting them the company. The buyer is not purchasing intelligence. The buyer is purchasing a bounded execution model and evidence that the boundary was applied.

13. Non-goals

  • Not an agent framework. The system does not decide how agents plan, prompt, or choose models.
  • Not a model provider. The boundary should be independent of model vendor.
  • Not a container platform. It may use container mechanisms, but image distribution and orchestration are not the product.
  • Not an observability dashboard. Observation after damage is insufficient.
  • Not a claim of perfect security. The claim is explicit authority reduction under stated assumptions.

14. Roadmap

The implementation should move from semantics to enforcement. A weak but clear boundary can be strengthened. A vague boundary cannot be audited.
